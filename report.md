# Проверка модуля конфигурации `pvs-js`

Проверка выполнена для `pvs-js 7.42.122+486086bb`.

Тесты реализованы как black-box сценарии в `PvsJsCliSpecificationTest`: каждый кейс разворачивает собственное дерево файлов во временной директории, запускает внешний `pvs-js`, затем проверяет код возврата, `stderr` и `PVS-Studio.json`.

## 1. Кейсы.

### 1.1. Невалидный корневой `pvs-settings.toml`

- **Требование:** спецификация требует поддерживать парсинг и валидацию `TOML`, а ошибки конфигурации должны быть наблюдаемы через код возврата и диагностический вывод.
- **Что проверяем:** синтаксически неверный корневой `pvs-settings.toml` должен остановить анализ и не дать построить отчёт.
- **Тестовые данные:** в корне проекта лежит `pvs-settings.toml` со значением `!!!`; анализируемый файл `src/test.js` содержит `const value = 1;`.
- **Как воспроизвести:** создать дерево
  - `pvs-settings.toml` -> `!!!`
  - `src/test.js` -> `const value = 1;`
  затем выполнить `pvs-js analyze .`
- **Фактический результат:** `exit code = 2`, в `stderr` есть путь к `pvs-settings.toml` и текст `Unknown token`, файл `PVS-Studio.json` не создаётся.
- **Вывод:** соответствует. Ошибка парсинга корневого `TOML` корректно останавливает запуск.

### 1.2. Дублирующийся ключ в `pvs-settings.toml`

- **Требование:** спецификация требует валидировать `TOML`; дублирование ключей в конфигурации должно трактоваться как ошибка входных данных.
- **Что проверяем:** корневой конфиг с двумя одинаковыми ключами не должен приниматься.
- **Тестовые данные:** в корне проекта лежит `pvs-settings.toml` со строками `threads = 1` и `threads = 2`; анализируемый файл `src/test.js` содержит `const value = 1;`.
- **Как воспроизвести:** создать дерево
  - `pvs-settings.toml` -> `threads = 1` и `threads = 2`
  - `src/test.js` -> `const value = 1;`
  затем выполнить `pvs-js analyze .`
- **Фактический результат:** `exit code = 2`, в `stderr` есть путь к `pvs-settings.toml` и текст `Duplicate key`, файл `PVS-Studio.json` не создаётся.
- **Вывод:** соответствует. Дублирующийся ключ корректно распознаётся как невалидный `TOML`.

### 1.3. Неизвестный ключ верхнего уровня

- **Требование:** спецификация отдельно выделяет поведение для неизвестных или лишних ключей: они не должны теряться незаметно без осмысленной политики обработки.
- **Что проверяем:** как `pvs-js` ведёт себя при наличии неизвестного ключа в корневом `pvs-settings.toml`.
- **Тестовые данные:** в корне проекта лежит `pvs-settings.toml` со строкой `unknown = 1`; анализируемый файл `src/test.js` содержит `const value = 1;`.
- **Как воспроизвести:** создать дерево
  - `pvs-settings.toml` -> `unknown = 1`
  - `src/test.js` -> `const value = 1;`
  затем выполнить `pvs-js analyze .`
- **Фактический результат:** `exit code = 0`, `stderr` пустой, `PVS-Studio.json` создаётся, в отчёте `warnings` пуст.
- **Вывод:** соответствует частично. Фактическая политика анализатора состоит в молчаливом игнорировании неизвестного ключа; без текста исходной спецификации нельзя утверждать, что именно такое поведение требовалось.

### 1.4. Поведение `.PVS-Studio/additional.rules.toml` при default discovery

- **Требование:** спецификация описывает иерархию уровней конфигурации и требует проверять, какие уровни реально участвуют в построении эффективной конфигурации.
- **Что проверяем:** учитывается ли `.PVS-Studio/additional.rules.toml` при обычном `analyze <dir>` и приводит ли его ошибка к ошибке конфигурации.
- **Тестовые данные:** в корне проекта лежит `.PVS-Studio/additional.rules.toml` со значением `!!!`; анализируемый файл `src/test.js` содержит `const value = 1;`.
- **Как воспроизвести:** создать дерево
  - `.PVS-Studio/additional.rules.toml` -> `!!!`
  - `src/test.js` -> `const value = 1;`
  затем выполнить `pvs-js analyze .`
- **Фактический результат:** `exit code = 0`, `stderr` пустой, `PVS-Studio.json` создаётся, ошибка парсинга не возникает.
- **Вывод:** не соответствует ожидаемой модели проверки уровня конфигурации. На практике этот файл в сценарии default discovery не участвует в разборе и потому даже невалидный `TOML` не влияет на запуск.

### 1.5. Поведение вложенного `pvs-settings.toml` в поддиректории

- **Требование:** спецификация выделяет уровень поддиректории и его приоритет относительно проектного уровня.
- **Что проверяем:** парсится ли вложенный `pvs-settings.toml` автоматически при запуске анализа от корня проекта.
- **Тестовые данные:** в поддиректории `src/nested` лежит `pvs-settings.toml` со значением `!!!`; анализируемый файл `src/test.js` содержит `const value = 1;`.
- **Как воспроизвести:** создать дерево
  - `src/test.js` -> `const value = 1;`
  - `src/nested/pvs-settings.toml` -> `!!!`
  затем выполнить `pvs-js analyze .`
- **Фактический результат:** `exit code = 0`, `stderr` не содержит упоминания `src/nested/pvs-settings.toml` и не содержит `Unknown token`, `PVS-Studio.json` создаётся.
- **Вывод:** не соответствует ожидаемой модели уровня поддиректории. При анализе от корня вложенный конфиг автоматически не разбирается.

### 1.6. Приоритет CLI-переопределения `--rules`

- **Требование:** спецификация требует проверять приоритеты уровней и переопределения из командной строки.
- **Что проверяем:** аргумент CLI `--rules=V7003=off` должен побеждать базовую конфигурацию и менять эффективное поведение анализатора.
- **Тестовые данные:** анализируемый файл `src/test.js` содержит
  ```js
  const x = 1; if (x > 0) { } else if (x > 0) { }
  ```
  Этот пример стабильно порождает диагностическое сообщение `V7003` без переопределения.
- **Как воспроизвести:** создать `src/test.js` с указанным кодом и выполнить:
  - базовый запуск: `pvs-js analyze .`
  - запуск с переопределением: `pvs-js analyze . --rules=V7003=off`
- **Фактический результат:** без CLI-переопределения `exit code = 0`, в `PVS-Studio.json` есть предупреждение с `"code":"V7003"`; с `--rules=V7003=off` `exit code = 0`, предупреждение `V7003` исчезает, `warnings` становится пустым.
- **Вывод:** соответствует. CLI-переопределение имеет ожидаемый приоритет и меняет эффективную конфигурацию.
